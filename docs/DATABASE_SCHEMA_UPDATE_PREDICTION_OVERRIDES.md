# Database Schema Update: Prediction Overrides

**Version:** 1.0  
**Date:** July 2, 2026  
**Related Feature:** Manual Prediction Window Overrides

## Overview

This document describes the database schema changes required to support manual prediction window overrides in the Match Management module.

## Collection: `matches`

### New Field: `predictionOverride`

**Path:** `matches/{matchId}/predictionOverride`

**Type:** Object (Map)

**Purpose:** Stores manual override information when administrators manually control prediction open/close state, taking precedence over automatic scheduling.

**Nullable:** Yes (null or omitted means no override is active)

**Default Value:** `null` or `{ isActive: false }`

### Field Structure

```javascript
predictionOverride: {
  isActive: boolean,        // Required. Whether override is currently active
  status: string,           // Required. Override status (must be valid MATCH_STATUS)
  timestamp: Timestamp,     // Required. When override was created
  performedBy: string,      // Required. UID of administrator who created override
  reason: string            // Optional. Reason or note for audit purposes
}
```

### Field Specifications

#### `isActive`
- **Type:** Boolean
- **Required:** Yes (when predictionOverride object exists)
- **Default:** false
- **Description:** Indicates whether the override is currently active
- **Validation:** Must be boolean
- **Business Rule:** When false, automatic scheduling applies; when true, override takes precedence

#### `status`
- **Type:** String
- **Required:** Yes (when isActive is true)
- **Allowed Values:** 
  - `prediction_open` - Match predictions manually opened
  - `prediction_locked` - Match predictions manually closed
- **Description:** The manually set prediction status
- **Validation:** Must be one of the valid prediction-related statuses
- **Business Rule:** This status takes precedence over automatic calculation

#### `timestamp`
- **Type:** Firestore Timestamp
- **Required:** Yes (when isActive is true)
- **Timezone:** IST (Asia/Kolkata, GMT+05:30)
- **Description:** When the override was created
- **Generated By:** `serverTimestamp()` on creation
- **Business Rule:** Used for audit trail and display purposes

#### `performedBy`
- **Type:** String (UID)
- **Required:** Yes (when isActive is true)
- **Description:** User ID of the administrator who created the override
- **Validation:** Must be valid user UID with admin privileges
- **Relationship:** References `users/{uid}`
- **Business Rule:** Used for audit trail and accountability

#### `reason`
- **Type:** String
- **Required:** No
- **Max Length:** 500 characters
- **Description:** Optional explanation for why override was created
- **Default:** "Manual override by administrator"
- **Business Rule:** Displayed in UI and audit logs for transparency

## Backward Compatibility

### Reading Legacy Documents

**Strategy:** Null-safe access pattern

All code accessing `predictionOverride` uses safe navigation:

```javascript
const hasActiveOverride = match.predictionOverride?.isActive ?? false;
const overrideStatus = match.predictionOverride?.status;
```

**Result:** Legacy matches without `predictionOverride` field behave exactly as before, using automatic scheduling.

### Writing to Existing Matches

When updating existing matches:
- Field is only added when manual override action is triggered
- Automatic scheduling continues to work without the field
- No migration required

### Firestore Security Rules

Rules must handle missing field gracefully:

```javascript
// Validate predictionOverride structure when present
function isValidPredictionOverride(override) {
  return override == null || (
    override.keys().hasAll(['isActive', 'status', 'timestamp', 'performedBy']) &&
    override.isActive is bool &&
    override.status is string &&
    override.timestamp is timestamp &&
    override.performedBy is string
  );
}

match /matches/{matchId} {
  allow update: if request.auth != null 
    && isAdmin(request.auth.uid)
    && isValidPredictionOverride(request.resource.data.predictionOverride);
}
```

## Data Lifecycle

### Creation
Created when administrator triggers `OPEN_PREDICTIONS` or `CLOSE_PREDICTIONS` lifecycle action.

### Updates
Override can be modified by subsequent manual actions:
- Opening after closure creates new override
- Closing after opening updates override
- Going to LIVE or COMPLETED status may clear override (implementation choice)

### Deletion
Override can be explicitly cleared (future enhancement) or implicitly cleared when:
- Match transitions to COMPLETED status
- Match is archived
- Administrator explicitly restores automatic scheduling

### Retention
Persists for audit trail purposes even after match completion.

## Security Considerations

### Write Access
- Only authenticated administrators can write `predictionOverride`
- Validated by Firestore security rules
- Additional authorization checks in application code

### Audit Trail
- Every override creation logged to `audit_logs` collection
- Includes: action, entityType, entityId, details.manual=true
- Links to `performedBy` UID for accountability

### Data Integrity
- `timestamp` must be server-generated to prevent tampering
- `performedBy` must match authenticated user
- `status` must be valid MATCH_STATUS enum value

## Query Patterns

### Check if Match Has Active Override

```javascript
const hasOverride = match.predictionOverride?.isActive ?? false;
```

### Get Override Status

```javascript
const overrideStatus = match.predictionOverride?.isActive 
  ? match.predictionOverride.status 
  : null;
```

### Find Administrator Who Created Override

```javascript
const adminUid = match.predictionOverride?.performedBy;
if (adminUid) {
  const adminUser = await getUserById(adminUid);
}
```

### Check Override Age

```javascript
const overrideDate = match.predictionOverride?.timestamp?.toDate();
const hoursSinceOverride = overrideDate 
  ? (Date.now() - overrideDate.getTime()) / (1000 * 60 * 60)
  : null;
```

## Performance Impact

### Read Performance
- Minimal impact: field is part of match document
- No additional reads required
- Field size is small (~200-300 bytes)

### Write Performance
- Minimal impact: field written only on manual actions
- No cascading updates required
- No transactions needed

### Storage Impact
- ~0.3 KB per match with override
- Estimated: <1% increase in total matches collection size
- Negligible cost impact
